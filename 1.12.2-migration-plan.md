# Slimefun4 1.12.2 Migration Plan

## Current Progress

The migration is still in an early bootstrap stage.

What is already in place:

- `build.gradle.kts` has been pushed toward Java 8 compatibility.
- `XSeries` has already been added as a dependency.
- Several compatibility helper classes have been introduced or are being drafted under `city/norain/slimefun4/compatibillty`.
- A repository scan report already exists in [compatibility-scan-report.md](/D:/Slimefun4/compatibility-scan-report.md).

What is still blocking the migration right now:

- `compileJava` is still failing.
- The codebase still contains Java 10+ / 14+ / 16+ syntax, such as `var`, switch rules, and pattern matching in `instanceof`.
- The project still depends on many 1.13+ APIs, including `BlockData`, `NamespacedKey`, `PersistentDataContainer`, modern `Sound`, modern `PotionMeta`, and flattened `Material` names.
- The plugin is not yet at the point where it can be tested on a real 1.12.2 server.

In short: the work has entered the "forced downgrade + compatibility cleanup" phase, but it has not yet reached the "compiles on Java 8" milestone.

## Actual Status Assessment

Based on the current repository state, the migration progress is roughly:

- Build/toolchain downgrade: started
- Java 8 syntax cleanup: partially started
- 1.12.2 API compatibility layer: partially started
- Full project compile on Java 8: not achieved
- Runtime verification on 1.12.2: not started

If we describe it as a coarse percentage, it is closer to 20% than 50%.
The direction is correct, but the hardest work is still ahead.

## Main Blocking Items

### 1. Java language level cleanup

This is the first hard gate.

Examples already visible from the current compile errors:

- `UniversalMenu.java`: `var`
- `UniversalMenuPreset.java`: pattern matching in `instanceof`
- `CompatibilityUtil.java`: switch rules
- `SlimefunPoolExecutor.java`: pattern matching in `instanceof`

Before the project can truly target Java 8, these modern language features must be removed across the codebase.

### 2. Replace modern Bukkit APIs

The second hard gate is API compatibility.

The current scan report already shows several high-risk areas:

- `BlockData` and related block-state logic
- `NamespacedKey`
- `PersistentDataContainer`
- `Tag` / `Registry`
- modern `PotionMeta` APIs
- modern `Sound` enum names
- post-1.13 `Material` flattening

These cannot be solved by syntax cleanup alone. They need either:

- a compatibility wrapper, or
- a legacy implementation branch, or
- a feature downgrade on 1.12.2

### 3. Dependency compatibility verification

Even after source code cleanup, some libraries may still block Java 8 or 1.12.2 compatibility.

The main ones that still need verification are:

- `dough-api`
- `HikariCP`
- `postgresql`
- `unirest-java`
- `guizhanlib`

## Recommended Single Plan

### Phase 1: Recover a real Java 8 compile baseline

Goal:

- make `compileJava` pass with Java 8 syntax rules

Tasks:

- remove all `var`
- rewrite switch expressions / switch rules to classic switch
- rewrite pattern matching `instanceof`
- continue scanning for Java 9+ / 10+ / 14+ / 16+ syntax usage

Exit condition:

- source compilation no longer fails because of Java language level

### Phase 2: Establish a stable 1.12.2 compatibility layer

Goal:

- centralize legacy handling instead of scattering version checks

Tasks:

- complete `LegacyBlockDataCompat`
- complete `LegacyPersistentDataCompat`
- complete `LegacyTagCompat`
- complete `LegacyRegistryCompat`
- add or refine a version utility entry point
- keep `XSeries` focused on enum/name mapping only

Exit condition:

- modern API usage has a single compatibility entry where possible

### Phase 3: Replace high-frequency incompatible APIs

Goal:

- unblock the bulk of compilation and runtime compatibility issues

Priority order:

1. `BlockData` family
2. `PersistentDataContainer` / `NamespacedKey`
3. `Material` / `Sound` / `Potion` mappings
4. `Tag` / `Registry`
5. entity, attribute, and recipe edge cases

Exit condition:

- the main modules can compile against the legacy target without large clusters of direct modern API usage

### Phase 4: Dependency and packaging validation

Goal:

- ensure the artifact can really be built and loaded

Tasks:

- verify each runtime dependency against Java 8
- confirm shaded libraries remain loadable
- verify `plugin.yml` and metadata assumptions for legacy servers

Exit condition:

- build artifact is produced successfully

### Phase 5: Real server verification on 1.12.2

Goal:

- move from "can compile" to "can actually run"

Minimum validation set:

- plugin loads
- plugin enables without fatal errors
- core commands work
- basic menus open
- core block and machine interactions do not immediately crash

Exit condition:

- the plugin can complete a smoke test on 1.12.2

## Suggested Immediate Next Step

The next most effective step is not adding more plan variants.
It is to finish Phase 1 and make the project pass Java 8 source compilation first.

Only after that should we measure the remaining 1.12.2 API breakage again.
